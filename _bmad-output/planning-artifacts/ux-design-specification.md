# UX Design Specification Breakout_Reborn

**Author:** Dhlee  
**Date:** 2026-03-28

---

## Executive Summary

### Project Vision

`Breakout_Reborn`의 UX 목표는 브릭브레이커 장르를 모바일 세로 화면에 맞게 다시 설계해, 플레이어가 “생각보다 먼저 손이 반응하는” 경험을 만드는 것이다. 사용자는 설치 없이 바로 게임에 들어와 몇 초 안에 조작을 이해하고, 첫 플레이 안에서 손맛과 보상, 재도전 욕구를 모두 체감해야 한다.

### Target Players

- 짧은 시간에 몰입 가능한 모바일 캐주얼 액션을 원하는 플레이어
- 고전 브릭브레이커 감성을 좋아하지만 현대적인 UX를 기대하는 플레이어
- 복잡한 메뉴나 학습 없이 즉시 플레이 가능한 게임을 선호하는 사용자

### Key Design Challenges

- 한 손 조작 중에도 공 궤적이 항상 잘 보여야 한다.
- 작은 모바일 화면에서 HUD와 플레이 필드가 서로 방해하지 않아야 한다.
- 반복 플레이의 핵심이 콘텐츠 양이 아니라 손맛과 재시도 흐름이 되도록 설계해야 한다.
- 멀티볼, 폭발 벽돌, 아이템 드롭이 겹쳐도 시각적으로 혼란스럽지 않아야 한다.

### Design Opportunities

- 상대 드래그 조작을 통해 손가림 없는 차별화 UX를 만들 수 있다.
- 실패 후 즉시 재도전 흐름을 매우 빠르게 설계해 “한 판 더”를 자연스럽게 유도할 수 있다.
- 파괴 피드백과 보상 선택 순간을 크게 연출해 적은 콘텐츠량도 풍부하게 느끼게 할 수 있다.

## Core Player Experience

### Defining Experience

플레이어가 가장 자주 하는 행동은 패들을 미세하게 움직여 공의 반사각을 조절하는 것이다. UX의 성패는 이 행동이 얼마나 자연스럽고 억울하지 않게 느껴지느냐에 달려 있다. 플레이어는 조작 자체를 의식하지 않고, 벽돌 파괴와 콤보 유지에 집중할 수 있어야 한다.

### Platform Strategy

- 플랫폼: 모바일 웹 우선
- 화면 방향: 세로형 9:16 고정
- 입력 방식: 터치 기반 상대 드래그 + 탭
- 기본 세션 길이: 2분 30초 ~ 4분
- 재시도 시간: 실패 후 5초 이내

### Effortless Interactions

- 화면 하단 어디를 눌러도 패들 이동이 시작되어야 한다.
- 탭은 발사 또는 액티브 사용으로 해석되며, 플레이어가 UI 버튼을 찾지 않아도 된다.
- 아이템 습득, 스테이지 클리어, 재시작 흐름은 별도 학습 없이 이해 가능해야 한다.
- 플레이 영역에서 가장 중요한 시선은 항상 “공”으로 돌아오게 설계한다.

### Critical Success Moments

- 첫 10초 안에 “드래그로 움직이고 탭으로 시작한다”를 이해하는 순간
- 첫 연쇄 파괴가 터지며 화면이 강하게 반응하는 순간
- 첫 실패 후 거의 즉시 다시 시작할 수 있어 “끊기지 않는다”는 인상을 받는 순간
- 첫 보상 선택에서 다음 스테이지가 달라질 것이라는 기대를 얻는 순간

### Experience Principles

1. 손보다 공이 먼저 보여야 한다.
2. 정보는 적게, 피드백은 크게 보여준다.
3. 실패는 짧고, 재시도는 즉시 가능해야 한다.
4. 복잡한 시스템보다 체감되는 반응성이 우선이다.

## Emotional Response Design

### Desired Emotions

- 시작 직후: 가벼움, 즉시성, 부담 없음
- 첫 성공: “생각보다 잘 된다”는 자신감
- 콤보 구간: 몰입, 속도감, 과장된 통제감
- 실패 순간: 좌절보다 빠른 재도전 충동
- 보상 선택 순간: 짧지만 분명한 기대감

### Emotional Curve by Session

1. `0~10초`: 빠른 이해
2. `10~45초`: 첫 성공 경험
3. `45~120초`: 숙련과 긴장 증가
4. `120초 이후`: 혼란 직전의 카오스와 통제 쾌감
5. `실패 직후`: 손실감보다 재시작 욕구가 먼저 오도록 설계

### Frustration Prevention

- 공 궤적이 가려지지 않게 한다.
- 반사각이 예측 불가능하게 튀는 느낌을 제거한다.
- 멀티볼 상태에서 중요한 공을 잃어버리지 않게 시선 유도 장치를 둔다.
- 실패 연출은 짧게 유지해 좌절의 체류 시간을 줄인다.

## Interaction Model

### Touch Zones

- 입력 허용 구역: 화면 하단 35%
- 주 플레이 시선 구역: 화면 중앙 50%
- 정보 구역: 상단 15%

### Relative Drag Behavior

- 최초 터치 위치와 관계없이 패들 이동은 상대값으로 계산한다.
- 손가락이 좌우로 움직인 거리만큼 패들이 이동한다.
- 손가락이 화면 밖으로 벗어나면 패들은 즉시 정지한다.
- 감도 옵션은 `0.85 / 1.0 / 1.15` 세 단계만 허용한다.

### Tap Behavior

- 공 대기 상태: 탭 = 발사
- 액티브 아이템 준비 상태: 탭 = 아이템 사용
- 결과 화면 상태: 탭 = 기본 CTA 실행 금지

### Input Safeguards

- 탭과 드래그 상태가 동시에 충돌할 때는 현재 게임 상태 우선 규칙을 적용한다.
- 발사 직후 짧은 입력 보호 구간을 둬 의도치 않은 이중 입력을 막는다.
- 저프레임 상황에서도 입력은 누락보다 지연 최소화를 우선한다.

## Information Architecture

### HUD Hierarchy

상단 HUD는 아래 우선순위를 따른다.

1. 생명
2. 현재 스테이지
3. 콤보
4. 점수

원칙:

- 항상 한 줄 또는 두 개의 얕은 레이어 안에서 해결한다.
- 게임 중 플레이 필드 위에 큰 패널을 띄우지 않는다.
- 숫자는 빠르게 읽히는 굵은 스타일을 사용한다.

### In-Run Information

- 생명은 아이콘+숫자 조합으로 표시
- 콤보는 활성 상태일 때만 강조
- 점수는 항상 보이되 시각적 우선순위는 낮춤
- 보상 선택 전에는 다음 상태를 암시하는 짧은 전환 연출 제공

### Non-HUD Information

- 튜토리얼은 문장보다 짧은 키워드와 애니메이션으로 전달
- 아이템 획득은 플레이 필드 중심에서 짧게 알림
- 실패/클리어 결과는 화면 중앙에 짧고 강하게 표시

## Screen and State Design

### 1. Entry State

- 즉시 플레이 가능한 진입 화면
- CTA는 `시작` 하나를 최우선 배치
- 보조 옵션은 최소화

### 2. Pre-Launch State

- 공이 패들 위에 부착된 상태
- 드래그 연습 가능
- `탭해서 시작` 유도 애니메이션 표시

### 3. In-Run State

- 중앙에는 공, 벽돌, 이펙트만 유지
- 플레이 중 모달성 UI 금지
- 아이템 드롭은 공의 시선을 방해하지 않는 경로와 속도로 낙하

### 4. Reward Choice State

- 각 스테이지 클리어 후 1회 노출
- 선택지는 항상 2개 중 1개
- 문구는 짧고 효용이 즉시 이해되어야 함
- 체류 시간은 3초 내 판단 가능 수준을 목표로 설계

### 5. Failure State

- `실패 연출 -> 결과 표시 -> 재도전 가능` 순서
- 실패 연출 0.6초 이내
- 결과 UI 노출 1.5초 이내
- 기본 포커스는 `한 판 더`

### 6. Success / Clear State

- 짧고 보상감 있는 클리어 표현
- 곧바로 보상 선택 또는 다음 스테이지 진입으로 연결
- 긴 결과 화면 금지

### 7. Interrupted / Resume State

- 브라우저 포커스 이탈, 탭 전환, 알림 등 외부 인터럽트가 발생하면 즉시 일시정지 상태로 전환한다.
- 복귀 시에는 자동 재개하지 않고 `재개` CTA를 짧게 노출한다.
- 중단 상태에서는 드래그 입력을 모두 해제하고, 재개 후 첫 입력부터 다시 계산한다.

### 8. Edge-case State Rules

- 멀티볼 상태에서 보조 공만 남아도 런은 유지된다.
- 실패 판정과 보상 노출이 동시에 가능할 때는 실패 상태가 항상 우선한다.
- 아이템 획득 애니메이션 중 실패가 발생하면 애니메이션은 즉시 종료하고 실패 흐름으로 전환한다.
- 결과 오버레이 노출 중 일반 탭 입력은 무시하고 CTA 입력만 받는다.

## Visual Foundation

### Visual Priorities

1. 공의 위치와 속도
2. 충돌 지점
3. 파괴 결과
4. 위험 요소
5. 보상 선택

### Playfield Readability Rules

- 배경은 벽돌과 공보다 후퇴해야 한다.
- 폭발/파티클은 화려하되 공을 덮지 않아야 한다.
- 멀티볼 시에는 메인 공을 항상 가장 밝고 큰 하이라이트로 표현한다.
- 메인 공은 기본 공 또는 가장 최근에 패들과 직접 상호작용한 공으로 정의한다.
- 보조 공은 메인 공보다 낮은 발광 강도와 짧은 잔상을 사용한다.
- 강철 벽돌은 체력 단계를 색/균열/테두리로 동시에 표현

### Motion Rules

- 히트 스톱은 짧고 강하게
- 화면 흔들림은 보스급 연출이 아니라도 미세하게 사용 가능
- 연쇄 파괴 시 이펙트의 강도는 콤보 수와 함께 증가
- 장식용 모션은 줄이고, 정보 전달형 모션만 남긴다.

## UX Patterns

### Button Hierarchy

#### Primary CTA

- `시작`, `한 판 더`, `선택 완료`
- 크고 대비가 높아야 함
- 한 화면에 하나만 강조

#### Secondary CTA

- `종료`, `뒤로`, `나중에`
- 시각적 무게를 낮춘다

### Feedback Patterns

#### Success Feedback

- 파괴, 콤보 상승, 클리어에 짧고 강한 피드백 제공

#### Warning Feedback

- 마지막 생명, 공 추락 직전 위기 상태는 과하지 않게 경고

#### Error / Invalid Feedback

- 탭 불가 상태에서는 시각적 무반응 대신 아주 짧은 비활성 피드백 제공

### Modal / Overlay Patterns

- 런 중 전체 화면 모달 금지
- 선택 화면은 전환 상태에서만 노출
- 튜토리얼도 플레이를 완전히 막기보다 짧은 오버레이 우선

### Navigation Patterns

- 메인 흐름은 `진입 -> 플레이 -> 보상 -> 재도전`
- 깊은 메뉴 구조를 만들지 않는다
- 플레이 중 뒤로 가기 판단은 별도 확인을 최소화

### Loading / Empty States

- 모바일 웹 특성상 로딩이 생기면 짧은 진행 애니메이션 사용
- 빈 상태 문구보다 즉시 플레이 가능한 상태를 우선 설계

## Component Strategy

### Core Components

1. Top HUD Bar
2. Playfield Layer
3. Paddle Input Layer
4. Reward Choice Card
5. Result Overlay
6. Tutorial Hint Banner

### Component Rules

#### Top HUD Bar

- 최소 높이 유지
- 큰 숫자와 짧은 라벨
- 상시 표시

#### Reward Choice Card

- 2개 카드 병렬 구조
- 효과는 한 줄 요약 + 짧은 보조 설명
- 선택 즉시 반응

#### Result Overlay

- `한 판 더` 우선
- 실패 이유 설명 최소화
- 버튼 수 2개 이하

#### Ball Highlight Rule

- 메인 공은 외곽선, 밝기, 잔상 길이 모두 가장 강하게 표현한다.
- 보조 공은 동일 색 계열이되 채도와 두께를 낮춰 시각적 우선순위를 낮춘다.
- 멀티볼 상태에서 공끼리 겹치면 메인 공이 항상 위 레이어에 렌더링된다.

### Reuse Strategy

- 버튼, 카드, 결과 패널의 규칙을 반복 사용해 학습 비용을 줄인다.
- 같은 정보는 항상 같은 위치, 같은 색 의미, 같은 애니메이션 리듬으로 보여준다.

## User Journeys

### Journey 1: 첫 진입 플레이

1. 사용자가 게임에 진입한다.
2. 즉시 시작 CTA를 본다.
3. 프리런치 상태에서 드래그와 탭을 본능적으로 시험한다.
4. 첫 발사를 한다.
5. 첫 연쇄 파괴 또는 첫 실패를 경험한다.
6. 즉시 재도전 또는 다음 시도를 선택한다.

### Journey 2: 성공적 런 진행

1. 스테이지 1 클리어
2. 2개 보상 중 하나 선택
3. 스테이지 2~4에서 점점 더 복합적인 상호작용 경험
4. 런 종료 후 다시 도전할 이유 확보

### Journey 3: 실패 후 재도전

1. 마지막 공 또는 생명 소진
2. 짧은 실패 연출
3. 결과 오버레이 노출
4. `한 판 더` 클릭
5. 새 런 즉시 시작

### Journey 4: 중단 후 복귀

1. 플레이 중 외부 인터럽트 발생
2. 게임이 즉시 일시정지
3. 복귀 시 `재개` 버튼 노출
4. 플레이어가 재개를 누르면 입력 상태 초기화 후 게임 복귀

## UX Event Map

### Core UX Events

- `run_entered`: 런 진입 완료
- `launch_ready`: 발사 전 대기 상태
- `ball_launched`: 첫 발사 완료
- `combo_increased`: 콤보 상승
- `powerup_dropped`: 아이템 드롭 발생
- `powerup_collected`: 아이템 획득 완료
- `stage_cleared`: 스테이지 클리어
- `reward_choice_opened`: 보상 선택 화면 노출
- `life_lost`: 생명 손실
- `run_failed`: 런 실패 확정
- `retry_available`: 재도전 입력 가능 상태
- `run_restarted`: 새 런 시작
- `session_interrupted`: 외부 요인으로 일시정지
- `session_resumed`: 재개 완료

### Event UX Rules

- `ball_launched` 이후 0.5초 내에는 튜토리얼 힌트가 자동 사라진다.
- `run_failed` 발생 시 모든 보조 애니메이션보다 실패 오버레이가 우선한다.
- `reward_choice_opened` 동안 플레이 입력은 비활성화된다.
- `session_interrupted` 발생 시 드래그 상태는 즉시 초기화된다.

## 2026-03-29 Commercial UX Polish Amendments

본 개정은 `Samus Shepard` 중심 시니어 디자인 리뷰와 `Freya` UX 논의를 반영한 상용 품질 보강안이다. 목적은 현재 UX가 "정상 동작하는 프로토타입"을 넘어서, 실제 상품처럼 매끈하고 기억에 남는 감각을 제공하도록 기준을 높이는 것이다.

### UX Diagnosis

- 현재 UX는 기능 전달은 충분하지만 상품화된 감정 리듬은 약하다.
- HUD, 보상 카드, 실패 오버레이, 상태 피드백이 모두 안정적이지만 지나치게 설명적이다.
- 플레이어는 무엇을 해야 하는지는 이해하지만, 왜 더 하고 싶은지는 아직 약하게 느낄 수 있다.

### Commercial UX Principles

#### 1. Every Important Action Needs a Premium Response

중요한 플레이어 행동은 모두 "내 입력이 가치 있었다"는 반응을 돌려줘야 한다.

- 드래그: 패들만 움직이는 것이 아니라 미세한 inertia와 라이트 반응이 따라온다.
- 탭 발사: 공이 떠나는 순간 launch flash, 짧은 음향 강조, 패들 recoil이 동반된다.
- 보상 선택: 카드 선택 후 선택된 카드가 잠깐 우세하게 확대되고 다음 장면으로 흐른다.
- 재도전: 단순 상태 리셋이 아니라 리듬감 있는 snap-back 전환을 사용한다.

#### 2. HUD Must Feel Engineered, Not Temporary

HUD는 정보판이 아니라 게임 브랜드의 일부처럼 느껴져야 한다.

- 상단 HUD는 더 강한 숫자 서체, 명확한 정렬, stage별 accent 색을 가진다.
- 콤보는 평시엔 절제되게, `emphasis/peak` 단계에서만 강하게 살아난다.
- 파워업 상태 표시는 단순 텍스트가 아니라 아이콘 + 짧은 상태 바 + 남은 charge/지속감 표현을 가진다.

#### 3. Reward Choice Must Feel Desirable

보상 카드 UX는 "설명을 읽는 시간"보다 "선택하고 싶은 충동"을 우선한다.

- 카드 구성: 아이콘, 2~4단어 헤드라인, 한 줄 효과, 즉시 체감 포인트
- 카드 반응: hover/pulse/선택 시 강조 애니메이션
- 선택 후 반응: 선택된 카드의 강한 확인, 비선택 카드의 부드러운 퇴장
- 카드 문구는 설명형보다 강화형 표현을 사용한다
  - 예: `패들 확장`보다 `와이드 패들`
  - 예: `레이저 충전`보다 `듀얼 레이저`

### Screen-Specific Amendments

#### Pre-launch

- `Drag, then tap to launch`는 장기적으로 단순 문구가 아니라 짧은 유도 모션과 함께 제공되어야 한다.
- 패들 위 공 부착 상태는 더 분명한 magnetic hover 느낌을 가져야 한다.
- 대기 상태가 정적이어서는 안 되며, 낮은 강도의 breathing motion을 가진다.

#### In-Run

- 브릭 파괴 시 숫자와 이펙트가 동시에 떠도 정보 혼잡이 생기지 않도록 중앙 시선축을 유지한다.
- 멀티볼 상황에서 메인 공, 패들, 위험 구간의 우선순위가 언제나 유지되어야 한다.
- 레이저/폭발/콤보 피크가 겹칠 때 HUD는 축소가 아니라 정리된 강조를 제공해야 한다.

#### Failure / Result

- 마지막 공 손실 직후 0.2초 전후의 감정적 끊김을 추가한다.
- 결과 화면은 벌점 통지가 아니라 "다시 할 이유"를 강조해야 한다.
- `다시 하기`는 가장 큰 버튼이며, stage 도달/최고 콤보/얻은 보상 같은 짧은 요약이 함께 보인다.

### Motion and Feedback Tokens

다음 UX 토큰은 공통 규칙으로 유지한다.

- `launch_pop`: 발사 순간의 짧은 팝과 반동
- `reward_reveal`: 카드 등장 시 아래에서 올라오며 살짝 멈추는 리듬
- `combo_peak`: peak 도달 시 숫자 확대, 색 강조, 주변 HUD dim
- `fail_drop`: 실패 시 중심부 수축과 CTA 잔존
- `split_flash`: 멀티볼 분열 시 공 중심 광량 증가와 분기
- `laser_trace`: 레이저 사용 시 선행 예고선과 잔광 잔상

### Anti-Prototype UX Constraints

다음 표현은 프로토타입 감각을 강화하므로 지양한다.

- 정지된 큰 안내 문구만으로 상태를 설명하는 구성
- 아이콘 없이 텍스트만 표시되는 파워업 HUD
- 항상 같은 강도로만 반응하는 콤보 표시
- 실패/보상/클리어 오버레이가 모두 동일한 패널 모양을 공유하는 구성

### UX Implementation Follow-up

- 보상 카드 시안은 별도 상품형 카드 규격으로 재정의할 것
- HUD 숫자/라벨/아이콘 체계를 하나의 visual token 세트로 통합할 것
- stage별 accent 색과 motion 강도 차이를 UX 규칙으로 명시할 것
- pre-launch, launch, reward, fail, retry의 핵심 UX 순간을 짧은 storyboard 수준으로 문서화할 것
